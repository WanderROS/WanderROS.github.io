SpringBoot Swagger2 Quick Start

发表于 2020-05-26 | 分类于 SpringBoot

简介

无论是前端开发还是后端开发，都或多或少地被接口文档折磨过!前端经常抱怨后端给的接口文档与实际情况不一致，后端又觉得编写及维护接口文档会耗费不少精力，经常来不及更新。其实无论是前端调用后端，还是后端调用后端，都期望有一个好的接口文档，但仅仅只通过强制来规范大家是不够的，随着时间推移，版本迭代，接口文档往往很容易跟不上代码更新的速度。Swagger的出现很好的解决了这一个问题，而且支持SpringBoot，只要使用注解就能很方便的为前端提供友好的API接口！这部分介绍如何使用SpringBoot快速构建带有Swagger接口的项目！

过去的文档迭代过程主要有两种：

开始

基础知识

  1. Swagger 提供了一套通过代码和注解自动生成文档的方法，这一点对于保证 API 文档的及时性将有很大的帮助。
  2. Swagger 是一套基于 OpenAPI 规范构建的开源工具，可以帮助我们设计、构建、记录以及使用 Rest API。Swagger 主要包含了以下三个部分：
    • Swagger Editor：基于浏览器的编辑器，可以用来编写 OpenAPI 规范
    • Swagger UI：将编写好的 OpenAPI 规范呈现为交互式的 API 文档
    • Swagger Codegen：可以通过为 OpenAPI规范定义的任何 API 生成服务器存根和客户端 SDK 来简化构建过程
  3. Swagger的优点：
    • 代码变，文档变。只需要少量的注解，Swagger 就可以根据代码自动生成 API 文档，很好的保证了文档的时效性
    • 跨语言性，支持 40 多种语言
    • Swagger UI 呈现出来的是一份可交互式的 API 文档，可以直接在文档页面尝试 API 的调用，省去了准备复杂的调用参数的过程
    • 可以将文档规范导入相关的工具（例如 SoapUI）, 这些工具将会为我们自动地创建自动化测试
  4. OAS是OpenAPI Specification的简称，可以翻译为OpenAPI规范，它是定义API的一种规范，它的前身是Swagger规范。
  5. 官网：https://swagger.io/
  6. 官方文档：http://springfox.github.io/springfox/docs/snapshot/
  7. 第三方UI页面：
    • https://github.com/caspar-chen/swagger-ui-layer
    • https://github.com/battcn/swagger-spring-boot

Swagger注解说明

@Api

  1. 用来设置对控制器的描述，修饰整个类，用于Controller类上
  2. tags=”说明该类的作用，可以在UI界面上看到的注解”
  3. value=”该参数没什么意义，在UI界面上也看到，所以不需要配置”

@ApiOperation

  1. 用来设置对接口的描述，用在Controller类的方法上
  2. value=”说明方法的用途、作用”
  3. notes=”方法的备注说明”

@ApiIgnore

  1. Swagger 文档不会显示拥有该注解的接口

@ApiImplicitParams & @ApiImplicitParam

  1. @ApiImplicitParams用于描述接口的非对象参数集

  2. @ApiImplicitParam用于描述接口的非对象参数

  3. 可以配置参数的中文含义，也可以给参数设置默认值，这样在接口测试的时候可以避免手动输入

  4. 有多个参数，则需要使用多个@ApiImplicitParam注解来描述，多个@ApiImplicitParam注解需要放在一个@ApiImplicitParams注解中

  5. @ApiImplicitParam注解中虽然可以指定参数是必填的，但是却不能代替@RequestParam(required = true)，必填只是在Swagger2框架内必填，抛弃了Swagger2，这个限制就没用了

    1. paramType的类型只有以下几种（参数放在哪个地方）:
      • header –> 请求参数的获取：@RequestHeader
      • query –> 请求参数的获取：@RequestParam

    3. name：参数名
    4. value：参数的汉字说明、解释

@ApiResponses & @ApiResponse

  1. @ApiResponses：用于请求的方法上，表示一组响应

  2. @ApiResponse：用在@ApiResponses中，一般用于表达一个错误的响应信息

  3. 参数：

    1. code：数字，例如400
    2. message：信息，例如”请求参数没填好”
    3. response：抛出异常的类

优缺点

优点

  1. 节省了大量手写接口文档的时间
  2. 通过注解自动生成在线文档
  3. 接口在线调用调试

缺点

  1. 代码耦合，需要注解支持
  2. 代码侵入性比较强
  3. 无法测试错误的请求方式、参数等

手动写API文档的痛点

  1. 文档更新交流不及时
  2. 接口返回结果不明确
  3. 不能直接在线测试接口，通常需要使用工具，比如postman
  4. 接口文档太多，不好管理

问题

  • 没有编写Swagger配置类，会出现错误
  • 如果paramType与方法参数获取使用的注解不一致，会直接影响到参数的接收
  • API列表中同一个方法出现多个，需要在RequestMapping中指定method类型

总结

体上来说，Swagger的配置还是比较简单的，并且Swagger能够自动帮我们生成文档确实为我们节省了不少工作，对后续的维护也提供了很大的帮助。除此之外，Swagger还能根据配置自动为我们生成测试的数据，并且提供对应的HTTP方法，这对我们的自测和联调工作也有不少的帮助，所以我还是推荐大家在日常的开发中去使用Swagger，应该可以帮助大家在一定程度上提高工作效率。Springfox-Swagger2可以通过扫描代码去生成描述文件，这样连描述文件都不需要再去维护了，所有的信息都在代码里面了！做到了代码即接口文档，接口文档即代码。在项目里使用这个能够大大方便自己的开发！希望以后多多使用，不要忘记了接口类还有这么一个好的工具！
